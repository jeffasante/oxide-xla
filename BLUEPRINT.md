# OxideXLA -- Project Blueprint

> **A high-speed Rust compiler for transforming PyTorch FX graphs into pure, stateless JAX functions.**
> Load any ONNX model. Get runnable JAX.

**Author:** Jeffrey Asante
**License:** Apache 2.0
**Language:** Rust (core) + Python (FX bridge)
**Status:** Pre-development

---

## The Problem

The PyTorch → JAX migration path is broken. The options today are:

| Approach | Why It Fails |
|----------|-------------|
| **Runtime wrappers** (`pytorch2jax`) | Calls PyTorch ops inside JAX — no real transpilation, no XLA benefits |
| **Ivy** | Unstable, effectively abandoned, not production-grade |
| **Manual rewrite** | The only reliable path — costs weeks per model |

Nobody has built a **graph-level transpiler** that takes ONNX (the universal ML interchange format) and emits idiomatic, XLA-compilable JAX code.

**OxideXLA fills that gap.**

## Architecture

```
PyTorch nn.Module
     |  (torch.onnx.export / FX -> ONNX shim)
     v
  .onnx file
     |
     v
+-------------------------------------------------+
|                OxideXLA (Rust)                   |
|                                                 |
|  [parser]     -- parse protobuf -> typed IR      |
|       |                                         |
|  [graph]      -- topological sort,              |
|       |         shape inference                  |
|       |                                         |
|  [op mapper]  -- ONNX op -> JAX equivalent       |
|       |                                         |
|  [codegen]    -- emit .py with @jax.jit          |
|                                                 |
+-------------------------------------------------+
     |
     v
  output.py  ->  XLA compile  ->  hardware
```

## Target Output Format — Pure JAX

The generated code is **stateless and functional**. Parameters are separated from computation — proving mastery of the JAX paradigm:

```python
# Generated by onnx-jax
import jax
import jax.numpy as jnp

@jax.jit
def forward(params, x):
    # onnx-jax mapped MatMul → jnp.matmul, Add → jnp.add
    x = jnp.matmul(x, params['layer1']['weights']) + params['layer1']['bias']
    x = jax.nn.relu(x)
    return x
```

> [!NOTE]
> Pure JAX output means **zero framework dependencies** — no Flax, no Equinox required. The output is directly `jax.jit` and `jax.grad` compatible, making it optimization-ready out of the box.

## Build Stages

Each stage ships independently. Each one is a portfolio artifact.

### Stage 1 — ONNX Graph Inspector CLI

> **Goal:** Load any `.onnx` file and pretty-print the full compute graph.

- Parse ONNX protobuf into typed Rust IR
- Display: nodes, op types, tensor shapes, dtypes, input/output topology
- Two output modes: **ASCII** (human-readable) and **JSON** (machine-readable)
- **Timeframe:** 1–2 weeks
- **Key skills:** `onnx-ir`, protobuf, Rust CLI (`clap`)

**Done when:**
- [x] `onnx-jax inspect resnet18.onnx` prints full graph with shapes
- [x] ASCII and `--json` modes both work
- [x] Published to crates.io
- [x] README has a one-command demo GIF

---

### Stage 2 — ONNX → JAX Codegen

> **Goal:** Emit runnable JAX Python for a defined op subset.

- Walk graph topologically
- Map each ONNX op to its JAX equivalent
- Perform static shape inference
- Write a self-contained `.py` module
- **Timeframe:** 3–4 weeks
- **Key skills:** Shape inference, codegen, JAX op mapping

**Done when:**
- [x] `onnx-jax compile mobilenet_v2.onnx --out model.py` produces a runnable file
- [x] `jax.jit(forward)(dummy_input)` completes without error on emitted code
- [x] Op coverage: MatMul, Add, Relu, Softmax, Reshape, Transpose, Conv (basic)
- [x] Integration test suite passes on 3 reference models

---

### Stage 3 — PyTorch FX → ONNX Bridge

> **Goal:** Close the full PyTorch → JAX loop.

- Python shim accepts a `torch.nn.Module`
- Traces with `torch.fx`, exports to ONNX
- Pipes result into Stage 2 Rust tool
- **Timeframe:** 1–2 weeks
- **Key skills:** `torch.fx`, Python FFI, subprocess/stdin piping

**Done when:**
- [x] `python bridge/fx_export.py --model torchvision.models.resnet18 | onnx-jax compile - --out r18.py`
- [x] PyTorch forward pass output matches JAX forward pass output within `1e-4`
- [x] Published as `pip install onnx-jax-bridge`

## Operator Coverage Plan

| ONNX Op | Input Signature | JAX Equivalent | Difficulty |
|---------|----------------|----------------|------------|
| `MatMul` | `A[m,k], B[k,n]` | `jnp.matmul(A, B)` | 🟢 Easy |
| `Relu` | `X[*]` | `jax.nn.relu(X)` | 🟢 Easy |
| `Softmax` | `X[n,c], axis` | `jax.nn.softmax(X, axis)` | 🟢 Easy |
| `Add` | `A[*], B[*]` | `jnp.add(A, B)` | 🟢 Easy |
| `Mul` | `A[*], B[*]` | `jnp.multiply(A, B)` | 🟢 Easy |
| `Reshape` | `X, shape[1D]` | `jnp.reshape(X, shape)` | 🟢 Easy |
| `Transpose` | `X, perm[]` | `jnp.transpose(X, perm)` | 🟢 Easy |
| `Conv` | `X, W, B, strides` | `lax.conv_general_dilated` | 🟡 Medium |
| `BatchNorm` | `X, scale, bias` | `flax.linen.BatchNorm` | 🟡 Medium |
| `LSTM` | `X, H, C, W` | `flax.linen.RNNCell` | 🔴 Hard |
| `Attention` | `Q, K, V, mask` | `jax.nn.dot_product_attention` | 🔴 Hard |

## Rust IR Design

```rust
/// Every ONNX op maps to a JaxOp variant
pub enum JaxOp {
    MatMul,
    Add,
    Mul,
    Relu,
    Softmax { axis: i64 },
    Reshape,
    Transpose { perm: Vec<usize> },
    Conv { strides: Vec<usize>, padding: String },
    BatchNorm { epsilon: f64, momentum: f64 },
}

/// A single node in the computation graph
pub struct Node {
    pub id: usize,
    pub op: JaxOp,
    pub name: String,
    pub inputs: Vec<usize>,          // IDs of preceding nodes
    pub output_shape: Vec<usize>,    // Statically inferred shape
    pub dtype: DType,
    pub param_key: Option<String>,   // e.g., "layer1/weights"
}
```

> [!IMPORTANT]
> The IR explicitly separates **data inputs** (tensors flowing through the graph) from **parameter inputs** (weights). This mirrors JAX's stateless paradigm where `params` are passed separately to the function.

## Final Objectives

### 🎯 Primary Objectives

1.  **Build the first open-source, graph-level ONNX → JAX transpiler**
    - Not a runtime wrapper — a true compiler that emits standalone JAX code
    - Output must be `jax.jit`-compilable and produce numerically identical results to the source model (within `1e-4` tolerance)

2.  **Publish to crates.io as a usable Rust library and CLI tool**
    - `cargo install oxide-xla` should give anyone a working transpiler
    - Clean API for programmatic use: `oxide_xla::compile("model.onnx", options)`

3.  **Close the PyTorch → JAX loop end-to-end**
    - A researcher should be able to go from `torchvision.models.resnet18()` to a running JAX function in a single command
    - No manual intervention, no model editing

4.  **Demonstrate production-grade Rust + ML infrastructure engineering**
    - Proper error handling, not panics
    - Comprehensive test suite (unit + integration)
    - CI/CD pipeline with automated testing against reference models

### 🚀 Stretch Objectives

5.  **WASM compilation**
    - Compile the Rust core to WebAssembly
    - Enable "transpilation in the browser" — paste an ONNX model, get JAX code
    - Demonstrates edge deployment capability

6.  **StableHLO backend**
    - Emit StableHLO instead of (or in addition to) JAX Python
    - Direct integration with OpenXLA ecosystem
    - Unlocks TPU-native deployment without Python

7.  **Operator fusion & optimization passes**
    - Detect fused patterns: `Conv + BatchNorm + ReLU` → single optimized JAX primitive
    - Algebraic simplification (identity removal, constant folding)
    - Dead code elimination

8.  **Performance benchmark suite**
    - Transpilation speed: target `< 50ms` for ResNet-18 (Rust advantage over Python tools)
    - Numerical accuracy: automated comparison of PyTorch vs JAX outputs
    - Published benchmark results in README

## Why This Gets Attention

This project sits at the intersection of three things ML infrastructure engineers care about **right now**:

- **JAX adoption is accelerating** across research labs, but PyTorch is where all the models live — the migration tooling gap is real and felt daily
- **Rust in ML infra** is a growing space (Candle, Burn, `ort`) and a Rust-native graph tool fits the pattern
- **ONNX is the universal format** — targeting it means the tool works on models from PyTorch, TensorFlow, and ONNX-native exporters

> [!TIP]
> The Stage 2 deliverable alone — an ONNX → JAX codegen in Rust — is a **publishable blog post** and a **real crates.io library**. Combined with the FX bridge it becomes the **first complete open-source PyTorch → JAX graph transpiler**.
